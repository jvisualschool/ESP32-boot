## 1단계: ROM 부트로더 (전원 켜자마자 실행)

칩에 전원이 들어오면, 공장에서 구워진 ROM 코드가 가장 먼저 실행됩니다. 이 부분은 절대 수정할 수 없습니다.

### `ESP-ROM:esp32s3-20210327`

이 칩에 내장된 ROM 코드의 버전입니다. 2021년 3월 27일에 만들어진 ESP32-S3용 ROM이라는 뜻입니다. ROM은 공장 출하 시 칩 내부에 영구적으로 기록된 프로그램으로, 전원이 들어오면 가장 먼저 실행되어 나머지 부팅 과정을 시작하는 역할을 합니다.

### `Build:Mar 27 2021`

위 ROM 코드가 빌드(컴파일)된 날짜입니다. 2021년 3월 27일입니다.

### `rst:0x15 (USB_UART_CHIP_RESET),boot:0x28 (SPI_FAST_FLASH_BOOT)`

두 가지 정보가 담겨 있습니다.

**rst:0x15 (USB_UART_CHIP_RESET)** — 이 칩이 왜 리셋되었는지를 알려줍니다. USB-UART 연결을 통해 리셋 신호가 들어왔다는 뜻입니다. PC에서 시리얼 모니터를 열거나 코드를 업로드할 때 자동으로 리셋이 걸리는데, 그 상황입니다.

**boot:0x28 (SPI_FAST_FLASH_BOOT)** — 부팅 모드를 알려줍니다. 외부 SPI 플래시 메모리에서 빠른 속도로 프로그램을 읽어서 부팅하겠다는 뜻입니다. 칩의 GPIO 핀 상태를 읽어서 부팅 모드를 결정하는데, 0x28이 일반 실행 모드에 해당합니다.

### `Saved PC:0x40377786`

리셋 직전에 CPU가 실행하고 있던 명령어의 메모리 주소입니다. PC는 Program Counter의 약자로, CPU가 "지금 어디를 실행 중인지" 가리키는 레지스터입니다. 디버깅할 때 "리셋 직전에 뭘 하고 있었나" 추적하는 용도입니다.

### `SPIWP:0xee`

SPI 플래시 메모리의 Write Protect 핀 상태 설정값입니다. 0xEE는 쓰기 보호가 비활성화된 상태를 의미합니다. 플래시 메모리에 데이터를 자유롭게 쓸 수 있다는 뜻입니다.

### `mode:DIO, clock div:1`

플래시 메모리와 통신하는 방식입니다.

**DIO (Dual I/O)** — 데이터를 2개 라인으로 동시에 주고받는 모드입니다. 1개 라인만 쓰는 SIO보다 빠르고, 4개 라인을 쓰는 QIO보다는 느립니다.

**clock div:1** — 클럭 분주비가 1이라는 뜻입니다. CPU 클럭을 나누지 않고 그대로 플래시 통신에 사용하므로 최고 속도로 동작합니다.

### `load:0x3fce2820,len:0x158c`

ROM 부트로더가 플래시에서 2단계 부트로더 코드 일부를 RAM 주소 0x3FCE2820에 복사합니다. 크기는 0x158C, 즉 10진수로 5,516바이트(약 5.4KB)입니다.

### `load:0x403c8700,len:0xd24`

2단계 부트로더의 또 다른 코드 조각을 RAM 주소 0x403C8700에 복사합니다. 크기는 0xD24, 즉 3,364바이트(약 3.3KB)입니다.

### `load:0x403cb700,len:0x2f34`

세 번째 코드 조각을 RAM 주소 0x403CB700에 복사합니다. 크기는 0x2F34, 즉 12,084바이트(약 11.8KB)입니다.

### `entry 0x403c8924`

RAM에 복사가 끝났으니, 이제 CPU의 실행 위치를 0x403C8924 주소로 점프시킵니다. 이 순간부터 ROM 부트로더의 역할은 끝나고, 2단계 부트로더가 실행을 시작합니다.

## 2단계: 2nd Stage Bootloader (ESP-IDF 부트로더)

ROM이 올려준 2단계 부트로더가 이제 본격적으로 시스템을 초기화합니다.

### `I (18) boot: ESP-IDF v5.5.2 2nd stage bootloader`

2단계 부트로더가 시작되었습니다. ESP-IDF v5.5.2 프레임워크로 빌드된 부트로더입니다. 숫자 18은 부팅 시작 후 18밀리초가 경과했다는 뜻입니다. 앞에 붙은 `I`는 Info 레벨 로그라는 표시입니다.

### `I (18) boot: compile time Feb  5 2026 14:16:34`

이 부트로더가 2026년 2월 5일 오후 2시 16분 34초에 컴파일되었다는 뜻입니다.

### `I (18) boot: Multicore bootloader`

멀티코어용 부트로더라는 뜻입니다. ESP32-S3는 CPU가 2개(PRO core, APP core)인데, 두 코어를 모두 활용하는 부트로더가 실행되고 있습니다.

### `I (18) boot: chip revision: v0.2`

이 칩의 실리콘 리비전이 v0.2라는 뜻입니다. 반도체는 설계 후 버그가 발견되면 수정된 버전을 다시 찍는데, 그 수정 버전 번호입니다. v0.2는 초기 생산분에 해당합니다.

### `I (21) boot: efuse block revision: v1.3`

eFuse 블록의 리비전입니다. eFuse는 칩 내부에 있는 일회성 기록 메모리로, MAC 주소, 보안 키, 칩 설정값 등이 공장에서 영구적으로 기록되어 있습니다. 그 데이터 구조의 버전이 v1.3이라는 뜻입니다.

### `I (25) boot.esp32s3: Boot SPI Speed : 80MHz`

플래시 메모리와의 SPI 통신 속도가 80MHz라는 뜻입니다. DIO 모드에서 80MHz면 이론상 최대 160Mbps(20MB/s)의 읽기 속도가 가능합니다.

### `I (28) boot.esp32s3: SPI Mode       : DIO`

Dual I/O 모드로 플래시를 읽고 있습니다.

### `I (32) boot.esp32s3: SPI Flash Size : 2MB`

플래시 메모리 크기가 2MB로 설정되어 있습니다. 이 값은 빌드 시 설정한 값입니다. (뒤에서 실제 칩은 16MB인데 2MB로 설정되어 있다는 경고가 나옵니다.)

### `I (36) boot: Enabling RNG early entropy source...`

난수 생성기(RNG)를 일찍 활성화합니다. 보안 통신(Wi-Fi, BLE 암호화 등)에는 예측 불가능한 난수가 필요한데, 부팅 초기부터 하드웨어 노이즈를 수집해서 엔트로피(무작위성)를 축적하기 시작하는 겁니다. 일찍 시작할수록 나중에 더 질 좋은 난수를 쓸 수 있습니다.

## 3단계: 파티션 테이블 읽기

플래시 메모리가 어떻게 나뉘어져 있는지 확인합니다. 리눅스에서 하드디스크를 파티션으로 나누는 것과 같은 개념입니다.

### `I (40) boot: Partition Table:`

파티션 테이블을 읽겠다는 선언입니다.

### `I (43) boot: ## Label            Usage          Type ST Offset   Length`

파티션 테이블의 컬럼 헤더입니다. 각 파티션의 이름(Label), 용도(Usage), 종류(Type), 하위 타입(ST), 시작 위치(Offset), 크기(Length)를 보여줍니다.

### `I (49) boot:  0 nvs              WiFi data        01 02 00009000 00006000`

**0번 파티션: nvs** — Non-Volatile Storage의 약자입니다. Wi-Fi SSID, 비밀번호, 사용자 설정값 등을 전원이 꺼져도 유지되도록 저장하는 영역입니다. 플래시 주소 0x9000부터 시작하고 크기는 0x6000(24KB)입니다.

### `I (56) boot:  1 phy_init         RF data          01 01 0000f000 00001000`

**1번 파티션: phy_init** — Wi-Fi/BLE 무선 통신 칩의 물리 계층(PHY) 초기화 데이터를 저장하는 영역입니다. 안테나 보정값, 송신 출력 설정 등이 들어 있습니다. 0xF000부터 시작하고 크기는 0x1000(4KB)입니다.

### `I (62) boot:  2 factory          factory app      00 00 00010000 00100000`

**2번 파티션: factory** — 실제 사용자 애플리케이션(ESP-Web-Monitor)이 저장된 영역입니다. 0x10000부터 시작하고 크기는 0x100000(1MB)입니다. 부트로더는 이 파티션에서 앱을 읽어서 실행합니다.

### `I (69) boot: End of partition table`

파티션 테이블 읽기가 끝났습니다.

## 4단계: 앱 이미지 로드

factory 파티션에서 실제 앱을 읽어 RAM에 올리는 과정입니다. 앱은 여러 개의 세그먼트(조각)로 나뉘어 저장되어 있고, 각각 다른 메모리 영역에 배치됩니다.

### `I (72) esp_image: segment 0: paddr=00010020 vaddr=3c020020 size=08670h ( 34416) map`

**세그먼트 0** — 플래시 물리 주소(paddr) 0x10020에서 읽어서, 가상 주소(vaddr) 0x3C020020에 **map** 합니다. 크기는 34,416바이트(약 33.6KB)입니다. map이란 RAM에 복사하지 않고 플래시를 직접 읽을 수 있도록 주소를 매핑하는 것입니다. 주로 읽기 전용 코드나 상수 데이터에 사용됩니다. 0x3C000000 대역은 플래시 캐시 영역입니다.

### `I (86) esp_image: segment 1: paddr=00018698 vaddr=3fc92000 size=02920h ( 10528) load`

**세그먼트 1** — 플래시에서 읽어 RAM 주소 0x3FC92000에 **load**(복사)합니다. 10,528바이트(약 10.3KB)입니다. 0x3FC00000 대역은 내부 SRAM 영역으로, 변수 등 읽기/쓰기가 필요한 데이터가 이곳에 올라갑니다.

### `I (89) esp_image: segment 2: paddr=0001afc0 vaddr=40374000 size=05058h ( 20568) load`

**세그먼트 2** — RAM 주소 0x40374000에 load합니다. 20,568바이트(약 20.1KB)입니다. 0x40000000 대역은 명령어 실행 영역(IRAM)으로, 빠르게 실행해야 하는 코드가 이곳에 올라갑니다.

### `I (99) esp_image: segment 3: paddr=00020020 vaddr=42000020 size=157ech ( 88044) map`

**세그먼트 3** — 가상 주소 0x42000020에 map합니다. 88,044바이트(약 86KB)로 가장 큰 세그먼트입니다. 0x42000000 대역은 플래시에서 직접 실행하는 코드 영역입니다. 앱의 주요 코드 대부분이 여기에 해당합니다.

### `I (117) esp_image: segment 4: paddr=00035814 vaddr=40379058 size=08f00h ( 36608) load`

**세그먼트 4** — IRAM 주소 0x40379058에 load합니다. 36,608바이트(약 35.8KB)입니다. 인터럽트 핸들러 등 지연 없이 실행해야 하는 코드가 이곳에 배치됩니다.

### `I (126) esp_image: segment 5: paddr=0003e71c vaddr=50000000 size=00020h (    32) load`

**세그먼트 5** — 주소 0x50000000에 load합니다. 단 32바이트입니다. 0x50000000 대역은 RTC 슬로우 메모리 영역으로, 딥 슬립(절전) 모드에서도 데이터가 유지되는 특수 메모리입니다.

### `I (132) boot: Loaded app from partition at offset 0x10000`

factory 파티션(오프셋 0x10000)에서 앱 로드가 완료되었습니다.

### `I (132) boot: Disabling RNG early entropy source...`

부팅 초기에 켜놨던 난수 생성기 엔트로피 소스를 끕니다. 충분한 엔트로피가 축적되었고, 이후에는 정상적인 RNG 경로를 사용합니다.

## 5단계: CPU 초기화 및 앱 시작

부트로더의 역할이 끝나고, 실제 애플리케이션 코드가 실행을 시작합니다.

### `I (143) cpu_start: Multicore app`

멀티코어 앱으로 시작합니다. 2개의 CPU 코어(PRO core = CPU0, APP core = CPU1)를 모두 사용하는 모드입니다.

### `I (152) cpu_start: GPIO 44 and 43 are used as console UART I/O pins`

GPIO 44번 핀은 UART TX(송신), GPIO 43번 핀은 UART RX(수신)로 사용됩니다. 지금 시리얼 모니터에서 이 메시지를 보고 있는 것 자체가 이 두 핀을 통해 데이터가 오가고 있다는 뜻입니다.

### `I (153) cpu_start: Pro cpu start user code`

PRO 코어(CPU0)가 사용자 코드 실행을 시작합니다. 여기서부터 사용자가 작성한 코드가 돌아갑니다.

### `I (153) cpu_start: cpu freq: 160000000 Hz`

CPU 클럭이 160,000,000Hz, 즉 160MHz로 설정되어 있습니다. ESP32-S3는 최대 240MHz까지 가능하지만, 현재 설정은 160MHz입니다. 전력 소모와 성능의 균형을 맞춘 설정입니다.

### `I (154) app_init: Application information:`

앱 정보를 출력하기 시작합니다.

### `I (158) app_init: Project name:     ESP-Web-Monitor`

프로젝트 이름이 **ESP-Web-Monitor**입니다.

### `I (163) app_init: App version:      1`

앱 버전 1입니다.

### `I (166) app_init: Compile time:     Feb  5 2026 15:28:25`

이 앱이 2026년 2월 5일 오후 3시 28분 25초에 컴파일되었습니다. 부트로더(14:16:34)보다 약 1시간 12분 뒤에 빌드된 것으로, 부트로더 먼저 빌드하고 앱을 나중에 빌드한 흐름입니다.

### `I (171) app_init: ELF file SHA256:  e0e0ba4fa...`

빌드된 ELF 바이너리 파일의 SHA256 해시값 앞부분입니다. 이 값으로 지금 실행 중인 바이너리가 어떤 빌드 결과물인지 정확히 식별할 수 있습니다. 같은 코드를 다시 빌드하면 다른 해시가 나오므로, 펌웨어 버전 추적에 사용됩니다.

### `I (175) app_init: ESP-IDF:          v5.5.2`

Espressif의 공식 개발 프레임워크 ESP-IDF 버전 5.5.2로 빌드되었습니다.

### `I (179) efuse_init: Min chip rev:     v0.0`

이 앱이 요구하는 최소 칩 리비전이 v0.0이라는 뜻입니다. 어떤 리비전의 ESP32-S3에서든 실행 가능합니다.

### `I (183) efuse_init: Max chip rev:     v0.99`

지원하는 최대 칩 리비전은 v0.99입니다. 사실상 현존하는 모든 ESP32-S3 리비전에서 동작합니다.

### `I (187) efuse_init: Chip rev:         v0.2`

현재 칩의 실제 리비전은 v0.2입니다. Min(v0.0) ~ Max(v0.99) 범위 안에 있으므로 정상 실행됩니다.

## 6단계: 힙 메모리 초기화

프로그램 실행 중 동적으로 메모리를 할당(malloc 등)할 수 있는 영역을 초기화합니다.

### `I (191) heap_init: Initializing. RAM available for dynamic allocation:`

동적 할당 가능한 RAM 영역을 초기화하겠다는 선언입니다.

### `I (197) heap_init: At 3FC95190 len 00054580 (337 KiB): RAM`

내부 SRAM 주소 0x3FC95190부터 337KB가 힙으로 사용 가능합니다. 가장 큰 영역이고 일반적인 변수, 버퍼, 객체가 여기에 할당됩니다.

### `I (202) heap_init: At 3FCE9710 len 00005724 (21 KiB): RAM`

추가 내부 SRAM 21KB입니다.

### `I (207) heap_init: At 3FCF0000 len 00008000 (32 KiB): DRAM`

전용 데이터 RAM(DRAM) 32KB입니다.

### `I (213) heap_init: At 600FE000 len 00001FE8 (7 KiB): RTCRAM`

RTC 메모리 영역 약 7KB입니다. 딥 슬립에서 깨어난 후에도 데이터가 유지되는 특수 메모리입니다.

**힙 합계: 337 + 21 + 32 + 7 = 약 397KB** 가 동적 할당에 사용 가능합니다.

## 7단계: SPI 플래시 감지

### `I (219) spi_flash: detected chip: boya`

플래시 메모리 칩 제조사가 **Boya Microelectronics**로 감지되었습니다. 중국 선전 소재의 플래시 메모리 전문 업체입니다.

### `I (221) spi_flash: flash io: dio`

플래시 I/O 모드가 DIO(Dual I/O)임을 다시 확인합니다.

### `W (224) spi_flash: Detected size(16384k) larger than the size in the binary image header(2048k). Using the size in the binary image header.`

**경고(W = Warning)** 입니다. 실제 장착된 플래시 칩은 **16,384KB(16MB)**인데, 빌드 설정에서는 **2,048KB(2MB)**로 되어 있습니다. 빌드 설정값(2MB)을 사용하겠다고 합니다. 즉 16MB 플래시 중 2MB만 사용하고 나머지 14MB는 놀고 있는 상태입니다. `menuconfig`에서 Flash Size를 16MB로 변경하면 전체를 활용할 수 있습니다.

## 8단계: 슬립 GPIO 설정

### `I (237) sleep_gpio: Configure to isolate all GPIO pins in sleep state`

절전(슬립) 모드 진입 시 모든 GPIO 핀을 격리(고임피던스) 상태로 설정합니다. 슬립 중 불필요한 전류 누설을 방지하기 위함입니다.

### `I (243) sleep_gpio: Enable automatic switching of GPIO sleep configuration`

슬립 모드 진입/복귀 시 GPIO 설정이 자동으로 전환되도록 활성화합니다.

## 9단계: 앱 메인 함수 실행

### `I (250) main_task: Started on CPU0`

메인 태스크가 CPU0(PRO core)에서 시작되었습니다. FreeRTOS 위에서 태스크로 실행됩니다.

### `I (260) main_task: Calling app_main()`

드디어 `app_main()` 함수를 호출합니다. C 프로그램의 `main()` 함수에 해당하는, 사용자가 작성한 코드의 진입점입니다. 전원 인가 후 약 260밀리초(0.26초)만에 여기까지 도달했습니다.

## 10단계: 사용자 앱 출력

앱이 출력한 첫 번째 메시지입니다.

### `This is esp32s3 chip with 2 CPU core(s), WiFi/BLE, silicon revision v0.2, 2MB external flash`

`esp_chip_info()` 함수로 읽어온 칩 정보를 출력한 것입니다. ESP32-S3, 듀얼코어, Wi-Fi/BLE 지원, 실리콘 리비전 v0.2, 외장 플래시 2MB(설정값 기준)입니다.

## 부팅 전체 흐름 요약

```
전원 ON (0ms)
  └→ ROM 부트로더 (칩 내장, 수정 불가)
       └→ 2nd Stage 부트로더 로드 (플래시 → RAM)
            └→ 파티션 테이블 읽기
                 └→ 앱 이미지 세그먼트 로드 (플래시 → RAM/캐시)
                      └→ CPU 초기화 (160MHz, 듀얼코어)
                           └→ 힙 메모리 초기화 (~397KB)
                                └→ FreeRTOS 스케줄러 시작
                                     └→ main_task → app_main() 호출 (260ms)
                                          └→ "ESP-Web-Monitor by Jinho Jung!"
```

전원 인가부터 사용자 코드 실행까지 **0.26초**입니다.

## 약어 및 용어 정리

| 약어/용어 | 풀네임 | 설명 |
|-----------|--------|------|
| **ROM** | Read-Only Memory | 공장에서 영구 기록된 메모리, 수정 불가 |
| **RAM** | Random Access Memory | 읽기/쓰기 가능한 휘발성 메모리 |
| **SRAM** | Static RAM | 전원 공급 중 데이터 유지, 리프레시 불필요, 빠름 |
| **DRAM** | Dynamic RAM | 주기적 리프레시 필요, SRAM보다 저렴하고 대용량 |
| **RTCRAM** | Real-Time Clock RAM | 딥 슬립 중에도 데이터가 유지되는 특수 메모리 |
| **SPI** | Serial Peripheral Interface | 직렬 통신 프로토콜, 플래시 메모리 연결에 사용 |
| **DIO** | Dual I/O | SPI 데이터 라인 2개를 사용하는 통신 모드 |
| **QIO** | Quad I/O | SPI 데이터 라인 4개를 사용하는 통신 모드 |
| **UART** | Universal Asynchronous Receiver/Transmitter | 비동기 직렬 통신 장치 |
| **GPIO** | General Purpose Input/Output | 범용 입출력 핀 |
| **FPU** | Floating Point Unit | 부동소수점 연산 전용 하드웨어 |
| **RNG** | Random Number Generator | 난수 생성기 |
| **eFuse** | Electronic Fuse | 일회성 프로그래밍 가능한 비휘발성 메모리 |
| **NVS** | Non-Volatile Storage | 전원 꺼져도 데이터 유지되는 저장 영역 |
| **PHY** | Physical Layer | 무선 통신의 물리 계층 |
| **ELF** | Executable and Linkable Format | 실행 파일 형식 (리눅스에서도 동일하게 사용) |
| **SHA256** | Secure Hash Algorithm 256-bit | 256비트 해시 함수, 데이터 무결성 검증용 |
| **IRAM** | Instruction RAM | 명령어 실행 전용 고속 RAM |
| **PC** | Program Counter | CPU가 현재 실행 중인 명령어 주소를 가리키는 레지스터 |
| **ESP-IDF** | Espressif IoT Development Framework | Espressif 공식 개발 프레임워크 |
| **FreeRTOS** | Free Real-Time Operating System | 오픈소스 실시간 운영체제 |
| **BLE** | Bluetooth Low Energy | 저전력 블루투스 통신 |
| **paddr** | Physical Address | 플래시 메모리의 실제 물리 주소 |
| **vaddr** | Virtual Address | CPU가 사용하는 가상 주소 |
| **KiB** | Kibibyte | 1,024바이트 (KB와 거의 동일하나 정확한 이진 단위) |
| **Hz** | Hertz | 주파수 단위, 1초당 진동/사이클 수 |
| **MHz** | Megahertz | 100만 Hz |
| **0x** | Hexadecimal prefix | 16진수 표기임을 나타내는 접두사 |
| **SIMD** | Single Instruction Multiple Data | 하나의 명령어로 여러 데이터를 동시 처리 |
| **CISC** | Complex Instruction Set Computer | 복잡한 명령어 집합 컴퓨터 |
| **RISC** | Reduced Instruction Set Computer | 축소 명령어 집합 컴퓨터 |
| **MMX** | Multi-Media Extensions | 인텔의 멀티미디어 SIMD 명령어 세트 |
| **MQTT** | Message Queuing Telemetry Transport | IoT용 경량 메시지 프로토콜 |
| **IoT** | Internet of Things | 사물인터넷 |
| **MCU** | Micro Controller Unit | 마이크로컨트롤러, 소형 단일칩 컴퓨터 |
